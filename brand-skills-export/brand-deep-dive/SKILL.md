# 品牌深扒 · 自动化情报收集

一句话：输入品牌名，输出七维情报报告。完全自动化，用户零手动。

## 技术路线

**底层方法：** 用 `browser_navigate` 访问 Bing 搜索 → scroll down → snapshot 提取搜索结果摘要 → 对有价值链接用 browser_navigate 直接访问获取详情。辅助：`execute_code` + Python `requests` 库做补充抓取。

**为什么不用 delegate_task + web_search？**
- ⛔ **关键陷阱：** `delegate_task` 子代理的 `web_search` 调用结果会被摘要层截断，实际搜索数据永远不会传回 orchestrator。这个路径已验证不可用（2026-05-27，连续6+次尝试全部失败）。
- 唯一有效路径是 browser_navigate 直接渲染搜索结果页面，从 snapshot 中提取内容。

**为什么不用浏览器直接上小红书？**
- API 全封（需要 X-s 签名）
- 云服务器 IP 被小红书标记为"IP at risk"，浏览器也打不开
- 搜索引擎缓存是唯一可行的数据源

## 执行流程

### 第一步：浏览器搜索引擎搜索

用 `browser_navigate` 直接访问 Bing CN 搜索结果页，一次一个 query，scroll + snapshot 提取结果。

**搜索策略（分批串行执行，共约6-8个query）：**

批次 A — 基础信息 + 口碑：
```
1. {品牌名} 品牌 创始人 公司 融资 创立
2. {品牌名} 小红书 评价 口碑 吐槽 2025 2026
3. {品牌名} 价格 值得买 性价比 测评
4. {品牌名} 竞品 对比 同类品牌
```

批次 B — 深度挖掘（基于批次A发现的线索调整关键词）：
```
5. "{具体创始人名}" "{公司名}" {品牌名} 融资
6. {品牌名} 争议 翻车 负面 投诉
7. {品牌名} 门店 销售 渠道 增长 2025
8. site:zhihu.com {品牌名} 体验 评价
```

**⚠️ Bing CN 去重陷阱：** Bing 国内版对不同 query 经常返回完全相同的一批结果（~10条固定结果）。这是搜索引擎的去重/缓存行为，不是数据真的只有这些。应对策略：
- 优先用 `&mkt=en-US&setlang=en` 参数尝试突破
- 交替使用中文和英文关键词
- 当遇到重复结果时，转用 `browser_navigate` 直接访问搜索结果中发现的高价值页面，提取详情
- 小众品牌可能确实只有这么多被索引内容，此时如实标注「⚠低可信度」即可

### 第二步：直接访问高价值页面

从搜索结果中识别出有价值的页面（非官网、非电商列表页），用 `browser_navigate` 直接访问：
- 百度百科页面 → 提取公司注册信息、创始人、成立时间
- 知乎文章 → 提取用户真实讨论
- 行业媒体（36kr/亿欧/猎云网）→ 提取融资、商业模式分析
- 国际时尚媒体 → 提取英文报道中的细节

遇到反爬拦截（DataDome、百度验证码等）时，记下URL和页面标题，从搜索结果摘要中提取已有信息。

**备选抓取方案：** `execute_code` + Python `requests` 对 Bing 搜索页做 HTML 解析（当 browser 遇到渲染问题时）。对中文内容站（百度百科、知乎、搜狐等），curl 大概率被反爬拦截，不浪费时间重试。

### 第二步：SEO 去噪（剔除品牌主动干预）

从所有搜索结果中，逐条判断并剔除以下品牌 SEO 污染：

**直接剔除（不计入有效结果）：**
| 污染类型 | 识别特征 | 处理 |
|---|---|---|
| 品牌官网 | 品牌官方域名（.com/.cn 等） | 剔除 |
| 品牌知识卡片 | Google/Bing 右侧面板 | 剔除 |
| SEO 通稿 | 标题含「品牌故事」「品牌介绍」「正式发布」「品牌理念」等 PR 腔，来自第三方域 | 剔除 |
| SEO 农场 | 内容农场、采集站、低质目录站、明显机器生成页 | 剔除 |
| 付费种草搬运 | 小红书笔记被 SEO 站完整转载，非原创 UGC | 剔除 |

**保留但降权：**
| 类型 | 识别特征 | 处理 |
|---|---|---|
| 疑似软文 | 独立域，标题含「评测」「测评」「怎么样」「值得买吗」 | 纳入但标注「⚠未验证独立性」 |

**单独保留：**
| 类型 | 识别特征 | 处理 |
|---|---|---|
| 品牌官网内容 | 品牌官方域下的「品牌故事」「品牌介绍」「关于我们」等 | ✅保留，备注「📌品牌官网发布」 |

> 去噪完成后，统计原始搜索结果数 与 去噪后有效独立结果数。

### 第三步：汇总输出

按七个维度整理，填入固定模板（见下方）。要求：**数据优先，不要空泛评价，每个结论跟一个数字或案例**。

数据可信度分级（与 brand-agency-intel 统一）：
- <20条 → ⚠低 | 20-60条 → ◐中 | 60-150条 → ✅较高 | >150条 → ✅✅高

## 使用方式

用户只需说品牌名，甚至不需要完整名称：

```
"扒一下珀莱雅"
"深扒花西子"
"研究一下观夏这个品牌"
```

如果不确定竞品，我会自动先搜确认。

## 已知限制

1. **数据不是实时 API 级别** — 依赖搜索引擎索引，热门品牌数据丰富，小众品牌可能信息不足
2. **没有精确搜索量数字** — 5118/千瓜的付费数据拿不到，只能从公开文章中提取估算值
3. **笔记数量是估算** — 来自第三方文章引用，不是实时 API 查询结果
4. **搜索结果有滞后** — 最新 1-2 天的内容可能没被索引
5. **中文品牌名搜索编码** — 部分搜索引擎（尤其Bing）会将多字中文品牌名拆解为单字搜索。解决方案：在搜索关键词前加英文品牌名（如 `"SkinCeuticals" 修丽可`）。详见 `brand-agency-intel` skill 的 `references/bing-search-pitfalls.md`。
6. **Bing CN 结果去重** — 已验证（2026-05-27）：Bing 国内版对不同 query 经常返回完全相同的 ~10 条固定结果，即使 keywords 完全不同。这是搜索引擎的缓存/指纹行为，不是数据就只有这些。应对：尝试 `&mkt=en-US` 参数 + 中英文交替关键词 + 直接访问搜索结果中的高价值页面。
7. **delegate_task + web_search 不可用** — 已验证（2026-05-27）：子代理虽然能调用 web_search，但返回的搜索结果在 orchestrator 的摘要层被截断，实际数据无法获取。不要使用此路径。
8. **中文内容站反爬严重** — 百度百科（验证码）、知乎（JSON渲染）、搜狐（404/鉴权）等站点对 headless 浏览器和 curl 请求有强反爬。优先从搜索引擎摘要中提取信息。
9. **国际时尚媒体可能被 DataDome 拦截** — Fashionista、WWD、Vogue Business 等站点使用 DataDome 反爬，browser_navigate 和 curl 均可能被拦。记录URL和标题，从搜索摘要提取关键信息。
10. **小众品牌数据极度稀疏** — 当品牌声量低时（如本 session 的 Fabrique），可能只获得品牌官网+百科+少量PR稿，缺乏真实UGC。此时报告应如实标注「⚠低可信度」，基于已有信息做推断（标注「推断」/「无数据」），不编造。
11. **🚫 综合洞察输出禁忌**：在"综合洞察"结论中，禁止使用"原单/尾货/仿品/跟单""品控不行""你们卖不动了"等触发品牌方防御的表述。原则：用"用户觉得"不用"产品有"，把负面包装成增量机会而非攻击短板。
